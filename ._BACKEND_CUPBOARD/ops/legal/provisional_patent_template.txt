 SECTION 5: DETAILED DESCRIPTION
============================================================

[Step-by-step operation. This is the most important section.
Be thorough — describe HOW each component works.]

5.1. WEARABLE FIRMWARE FLOW
  - Device boot and key generation
  - BLE GATT service advertisement
  - Panic trigger detection (button press, gesture, sensor)
  - CBOR payload construction:
    {
      "type": "panic",
      "ts": <unix_timestamp>,
      "lat": <latitude>,
      "lon": <longitude>,
      "dev": "<device_id>",
      "sig": "<ed25519_signature>"
    }
  - Payload signing with hardware keystore (SHA256withRSA or Ed25519)
  - BLE transmission to companion app
  - Companion app relay to backend verifier

5.2. DEVICE VERIFIER BACKEND
  - Receive signed payload
  - Verify signature against registered public key
  - Validate timestamp freshness (anti-replay)
  - Trigger operator alert chain
  - Log attestation for audit trail

5.3. TOKEN MINTING AND PPV PLAYBACK
  - Purchase flow via Stripe Connect Direct charge
  - Entitlement creation in database
  - JWT minting: RS256, claims (iss, sub, ent, evt, dev, iat, exp, jti, aud)
  - CDN signed URL generation with embedded token
  - Player loads manifest via signed URL
  - DRM license request with bearer token
  - License server validates entitlement and issues decryption key
  - Forensic watermark injection per session
  - Concurrency tracking and device binding

5.4. CORTEX ENGINE / AI
  - Data ingestion from fighter stats, event results, training data
  - Model architecture (describe at high level)
  - Coaching output generation
  - Feed ranking algorithm (factors, weights, signals)
  - Content curation pipeline

5.5. WEBXR PARITY
  - A-Frame scene construction
  - Asset loading and streaming
  - User interaction model
  - Integration with live event data

 SECTION 6: EXAMPLE IMPLEMENTATIONS
============================================================

EXAMPLE 1: LIVE PPV WITH CHUCKYA EMERGENCY
  - Fighter wears CHUCKYA device during event
  - Panic triggered -> signed CBOR payload -> verifier -> operator alert
  - Simultaneously, PPV viewers watch via token-gated stream
  - Forensic watermark identifies any leak source

EXAMPLE 2: TOKENIZED PLAYBACK FOR SMALL PROMOTER
  - Promoter creates event on DFC platform
  - Viewer purchases ticket (Stripe Direct charge on promoter account)
  - Backend mints JWT, generates CDN signed URL
  - Player loads encrypted stream, requests DRM license
  - Watermark embedded per viewer session

EXAMPLE 3: WEBXR ARENA EXPERIENCE
  - User opens WebXR link on mobile or VR headset
  - A-Frame loads neon octagon arena
  - Live fight data overlaid in real-time
  - Social features (chat, reactions) integrated
